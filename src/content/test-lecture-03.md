# 第3章：软件测试流程和规范

本章按 PPT“软件测试流程和规范”整理。考试重点：==测试左移/右移==、==W 模型与 TMap==、==敏捷测试与 Scrum/SBTM==、==测试流派==、==测试过程改进模型==、==测试标准与规范==。

## 2. 测试左移与测试右移

早期测试过程通常是：根据需求写用例和脚本，开始测试，提交 bug，回归测试，通过后结束。PPT 强调现代测试要左移和右移。

| 概念 | 含义 | 典型活动 |
| --- | --- | --- |
| 测试左移 | 向测试“开始”之前移动，更早发现不合理之处 | 产品需求文档评审、研发设计评审、单元测试、代码规范检查、代码复杂度检查 |
| 测试右移 | 向测试“结束”之后移动，上线后继续获得质量反馈 | 灰度发布、线上监控、用户反馈、混沌工程、A/B 测试 |

考试写法：

> 测试左移强调尽早发现问题，降低后期返工成本；测试右移强调上线后通过监控、反馈和实验持续评估质量。

## 3. 传统软件测试过程模型

### 3.1 W 模型

W 模型强调：测试过程和开发过程保持同步，互相依赖，但重点不同。项目启动的同时，测试工作也应该启动。

| 开发相关活动 | 对应测试活动 |
| --- | --- |
| 需求定义 | 需求评审、确认测试准备 |
| 设计 | 设计审查、测试计划、测试用例准备 |
| 编码 | 单元测试、集成测试 |
| 系统构建 | 系统测试 |
| 版本发布 | 版本审查、发布报告、已知问题清单 |

W 模型的核心不是多画一个 W，而是强调：==测试不应等编码完成才开始==。

### 3.2 测试工作像冰山

PPT 提到：明显可见的测试执行活动只是海面以上部分，平均约占测试活动 40%；计划和准备工作在海面以下，但对执行效率影响很大。

结论：

- 测试计划和准备越充分，测试执行越少占据项目关键路径。
- 不要把测试理解成“点点界面、跑跑用例”。
- 计划、准备、说明、执行、完成都属于测试生命周期。

## 4. TMap

TMap 是一种结构化的、基于风险策略的测试管理方法。目标是更早发现缺陷，以较小成本有效完成测试，减少发布后支持成本。

### 4.1 TMap 生命周期

| 阶段 | 主要内容 |
| --- | --- |
| 计划和控制 | 明确测试任务、范围、重点；基于风险制定策略、估算、计划；监控和调整测试过程 |
| 基础设施 | 准备测试环境、测试工具、测试件管理、缺陷管理、自动化框架等 |
| 准备 | 对测试依据进行可测试性评审，确保测试依据质量满足设计要求 |
| 说明 | 规划测试用例和前置条件，使测试对象交付后能尽快执行 |
| 执行 | 安装测试对象、预测试、执行脚本、验证结果、记录缺陷 |
| 完成 | 保存测试件，复用测试成果，进行过程评估和经验总结 |

### 4.2 TMap 三大基石

| 基石 | 含义 |
| --- | --- |
| L：测试活动生命周期 | 与软件开发生命周期一致，描述各阶段活动 |
| O：组织融合 | 测试小组融入项目组 |
| I：基础设施和工具 | 测试环境必须稳定、可控制、有代表性 |
| T：可用技术 | 支持测试过程的技术 |

PPT 中写 O、I、T 是 TMap 的基石，同时也强调生命周期 L。

### 4.3 TMap 优点

- 结构化测试过程清晰。
- 能够基于质量风险提供建议。
- 能够早期发现和预防缺陷。
- 能让测试执行尽量少占据关键路径。
- 测试脚本、测试用例等测试产品可以复用。
- 所有参与者使用同一套测试语言，促进标准化。

### 4.4 TMap 不必严格顺序执行

TMap 生命周期不是简单线性流程。计划、准备、说明等工作可以重叠或同步进行。只有执行阶段最明显地占据关键路径，其他阶段可以提前准备。

## 5. 敏捷测试

### 5.1 敏捷开发基本思想

PPT 提到敏捷宣言的四组价值观：

| 敏捷更重视 | 相对不那么重视 |
| --- | --- |
| 个体和互动 | 流程和工具 |
| 工作的软件 | 详尽的文档 |
| 客户合作 | 合同谈判 |
| 响应变化 | 遵循计划 |

注意：不是不要流程、工具、文档和计划，而是当两者冲突时，前者价值更高。

### 5.2 敏捷测试宣言

| 敏捷测试更重视 | 相对不那么重视 |
| --- | --- |
| 开发协作测试 | 测试分工和测试工具 |
| 可运行的测试脚本 | 写在纸上的测试用例 |
| 从客户角度理解需求 | 只从已定义需求判定结果 |
| 基于上下文及时调整测试策略 | 机械遵守测试计划 |

### 5.3 敏捷测试原则

- 尽早和持续开展测试。
- 基于风险的测试策略是必需的。
- 测试计划、设计、执行力求简单。
- 能及时完成对软件质量的全面评估。
- 软件本身是测试研究和分析的主要对象。
- 在满足质量前提下，测试越快越好。
- 对测试技术精益求精。
- 不断反思，持续优化测试流程和设计。

### 5.4 传统测试与敏捷测试区别

| 对比项 | 传统测试 | 敏捷测试 |
| --- | --- | --- |
| 流程 | 重流程，阶段性明确 | 持续测试，快速反馈 |
| 团队 | 测试团队相对独立 | 整个团队对质量负责 |
| 文档 | 关注需求和测试文档 | 更重视面对面沟通和可运行测试 |
| 计划 | 强调计划性和规范性 | 强调速度、适应性和上下文 |
| 技术 | 传统 V&V、过程改进 | TDD、ATDD、自动化、探索式测试 |

## 6. Scrum 与敏捷测试流程

Scrum 是一种迭代和增量的产品开发框架，允许持续反馈，应对需求频繁变化。任务划分为冲刺，每个冲刺不超过 1 个月。

### 6.1 Scrum 小组角色

| 角色 | 职责 |
| --- | --- |
| product owner | 关注业务，代表干系人和客户利益 |
| developers | 负责产品开发和支持 |
| scrum master | 指导团队理解和实践 Scrum |

### 6.2 Scrum 测试流程

| 阶段 | 测试关注 |
| --- | --- |
| 需求定义 | 确认测什么，估算工作量 |
| 迭代任务划分 | 明确每项任务完成标准 |
| 迭代实施 | 执行 TDD、单元测试、BVT |
| 敏捷验收 | 验证功能特性是否真正完成 |

敏捷验收测试与传统验收测试不同，更强调每个功能特性在迭代中是否达到完成标准。

## 7. SBTM：基于会话的测试管理

SBTM 用于管理探索式测试，尤其适合敏捷测试中新功能测试。

### 7.1 核心概念

| 概念 | 含义 |
| --- | --- |
| session | 一段不受打扰的测试时间，通常 90 分钟，是测试管理最小单元 |
| mission | 一个目标明确的测试任务 |
| charter | 每个 session 的测试指导，说明测什么、怎么测、找什么风险 |
| session sheet | 记录测试过程、结果、问题、缺陷和时间分布的报告 |

### 7.2 charter 应包含什么

- 测试目标。
- 测试范围。
- 测试策略。
- 要寻找的缺陷类型。
- 关注的质量风险。
- 参考资料、环境、测试数据。

### 7.3 session sheet 常见内容

| 内容 | 含义 |
| --- | --- |
| 测试执行者 | 谁执行了本次 session |
| TBS | Test/Bug/Setup 时间分布 |
| 测试数据 | 本次使用的数据和文件 |
| 测试笔记 | 为什么这样测、如何测、测到了什么 |
| 问题和风险 | 测试中的疑问和未来参考 |
| 缺陷 | 本次测试直接产出的 bug |

## 8. 软件测试流派

PPT 给出五大流派：

| 流派 | 观点 |
| --- | --- |
| 分析流派 | 测试是逻辑性的、技术性的，强调结构化测试和代码覆盖 |
| 标准流派 | 测试是可重复、可衡量进度的标准化工作 |
| 质量流派 | 测试是过程质量控制和项目风险度量活动 |
| 上下文驱动流派 | 强调人的能动性、启发式思维、探索式测试 |
| 敏捷流派 | 强调自动化测试，用测试快速验证开发是否完整，如 TDD |

考试注意：

- 分析流派偏技术和覆盖。
- 上下文驱动流派偏探索和人。
- 敏捷流派偏自动化和快速反馈。

## 9. 测试过程改进模型

### 9.1 TMMi

TMMi 描述测试过程成熟度。它吸收 CMM/CMMi 精华，基于测试过程历史演化和业界最佳实践。

常见考法：知道 TMMi 是测试成熟度模型，用来评估和改进测试过程能力。

### 9.2 TPI

TPI 是测试过程改进参考模型。

关键内容：

- 测试过程由多个关键域组成。
- PPT 给出 16 个关键域，如度量、缺陷管理、测试件管理、测试方法实践、测试用例设计、测试工具、测试环境、承诺、介入程度、测试策略、测试组织、沟通、报告、过程管理、估算和计划等。
- TPI 提供检查点和建议，帮助客观判断关键域级别，并指导改进。

### 9.3 TMap 与 TPI 区别

| 对比 | TMap | TPI |
| --- | --- | --- |
| 关注问题 | 测试应该如何做 | 什么样的测试是好的测试 |
| 性质 | 测试方法和过程指导 | 测试过程衡量和改进模型 |
| 内容 | 策略、计划、用例、执行、管理 | 关键域、级别、检查点、改进建议 |

### 9.4 CTP 与 STEP

| 模型 | 重点 |
| --- | --- |
| CTP | 关键测试过程，采用上下文相关方法，可以裁剪 |
| STEP | 系统化测试和评估过程，强调生命周期早期开始测试、基于需求测试、缺陷系统分析、测试人员和开发人员协作 |

## 10. 软件测试标准与规范

PPT 中列出的标准类别：

| 类别 | 例子 |
| --- | --- |
| 国际标准 | ISO 29119 系列 |
| 国家标准 | GB/T 15532-2008《计算机软件测试规范》、GB/T 9386-2008《计算机软件测试文档编制规范》 |
| 行业标准 | 证券期货、银行业、公安、电力、交通等行业测试规范 |
| 企业/机构规范 | 企业内部测试流程、模板、检查表 |
| 项目规范 | 某项目的测试准入、退出、度量、文档要求 |

### 10.1 完整测试规范包含什么

- 规范目的。
- 范围。
- 文档结构。
- 词汇表。
- 参考信息。
- 可追溯性。
- 方针。
- 过程/规范。
- 指南。
- 模板。
- 检查表。
- 培训和工具。

### 10.2 制定测试规范要考虑

| 内容 | 说明 |
| --- | --- |
| 角色 | 谁负责计划、设计、执行、评审、批准 |
| 进入准则 | 什么时候可以开始测试 |
| 输入项 | 需求、设计、代码、构建包、环境等 |
| 活动过程 | 每个阶段做什么 |
| 输出项 | 用例、报告、缺陷、总结 |
| 验证与确认 | 如何确认活动结果有效 |
| 退出准则 | 什么条件下可以结束测试 |
| 度量 | 覆盖率、缺陷趋势、执行进度、质量风险 |

## 11. 本章速记

1. 测试左移：更早发现问题；测试右移：上线后继续监控和反馈。
2. W 模型强调测试与开发同步。
3. TMap 是结构化、基于风险的测试管理方法。
4. 敏捷测试强调持续测试、快速反馈、全团队负责质量。
5. SBTM 用 session 管理探索式测试。
6. 五大测试流派：分析、标准、质量、上下文驱动、敏捷。
7. TMap 讲“怎么做测试”，TPI 讲“如何衡量和改进测试过程”。
8. 测试规范要有角色、进入准则、输入、过程、输出、退出准则和度量。
