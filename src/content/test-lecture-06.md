# 第6章：系统测试

本章按 PPT 整理。系统测试重点不是单个函数，而是完整系统与环境组合后是否满足需求。考试重点：==系统功能测试==、==接口/UI 自动化==、==回归测试==、==精准测试==。

## 1. 本章考点地图

| 模块 | 必须掌握 |
| --- | --- |
| 系统测试定义 | 集成测试后，把软件与硬件、数据、网络、支撑软件等结合测试 |
| 系统功能测试 | 功能、界面、数据、逻辑、接口 |
| 系统功能测试流程 | 需求分析、测试项、方法工具、用例、环境数据、执行、报告 |
| 接口测试 | HTTP、Web service、RESTful；验证接口和系统功能 |
| UI 测试 | 用户角色、业务路径、输入域和组合、异常操作 |
| 自动化测试 | 单元、接口、UI 的自动化层次 |
| Postman | 接口请求、collection、pre-request script、post-response script |
| Selenium/Cypress | Web UI 自动化工具 |
| Android UI 自动化 | Appium、Robotium、Espresso、Instrumentation 等 |
| 回归测试 | 修改后再次测试，发现回归缺陷 |
| 精准测试 | 建立代码和测试用例映射，基于变更选择回归用例 |

## 2. 系统测试定义

系统测试是在完成集成测试之后，将待测软件与计算机硬件、输入输出设备、数据、网络、支撑软件、第三方软件等综合在一起进行测试。

目标：

> 验证整个系统在功能、性能等方面满足用户预期和需求规格说明书。

系统测试的质量维度：

| 维度 | 对应测试 |
| --- | --- |
| 功能适应性 | 功能测试 |
| 兼容性 | 兼容性测试 |
| 性能 | 性能测试 |
| 安全性 | 安全测试 |
| 可靠性 | 可靠性测试 |
| 易用性 | 易用性测试 |

第 6 章主要讲系统功能测试，非功能测试在第 7 章详细讲。

## 3. 系统功能测试

功能测试可以发生在单元测试，也可以发生在系统测试。

| 层次 | 功能测试关注 |
| --- | --- |
| 单元测试中的功能测试 | 独立模块的功能是否正确，主要根据输入输出判断是否满足设计 |
| 系统测试中的功能测试 | 考虑模块间相互作用和系统应用环境，用规格说明书衡量，模拟用户完整业务 |

### 3.1 系统功能测试测什么

| 测试对象 | 内容 |
| --- | --- |
| 功能 | 每项功能是否符合实际要求 |
| 界面 UI | 界面是否清晰、美观，菜单和按钮是否正常 |
| 数据 | 输入、输出、保存、读取是否正确 |
| 逻辑 | 功能逻辑是否清楚，业务状态是否稳定变化 |
| 接口 API | 接口是否有效，返回是否正确，能否支撑系统功能 |

### 3.2 功能测试要点

- 每项功能符合实际要求。
- 界面清晰、美观。
- 菜单、按钮操作正常、灵活。
- 能处理异常操作。
- 能接受正确数据输入。
- 对异常输入有容错处理。
- 输出结果准确，格式清晰，可以保存和读取。
- 功能逻辑符合使用者习惯。
- 系统状态按业务流程变化并保持稳定。

### 3.3 系统功能测试基本思路

1. 明确质量要求和测试目标。
2. 阅读需求文档，分析被测功能，列出测试项。
3. 选择合适测试方法和工具。
4. 设计测试策略和测试用例。
5. 准备测试数据，搭建测试环境。
6. 执行测试用例。
7. 分析测试结果，评估测试充分性。
8. 提交测试报告。

系统规模通常较大，所以系统测试多采用黑盒测试技术，如等价类划分、边界值分析等。

## 4. 面向接口的功能测试

常见接口：

- HTTP 接口。
- Web service 接口。
- RESTful 接口。

接口测试通过调用软件接口模拟客户端向服务器发送请求，并验证是否获得正确返回信息。

注意：接口测试不只是验证接口能不能通，更重要的是通过接口验证系统功能。

### 4.1 HTTP 接口

| 报文 | 组成 |
| --- | --- |
| 请求报文 | 请求行、请求头部、空行、请求数据 |
| 响应报文 | 状态行、响应头部、空行、响应数据 |

### 4.2 Web service

PPT 术语：

- SOAP。
- WSDL。
- UDDI。

| 术语 | 含义 |
| --- | --- |
| SOAP | 基于 XML 进行信息交换的通信协议 |
| WSDL | 描述网络服务、函数、参数和返回值等访问信息 |
| UDDI | 描述、发布和寻找网络服务的机制 |

### 4.3 RESTful 接口

RESTful 是一种软件架构风格。

核心原则：

- 对象实例抽象为资源。
- 每个资源有唯一 URI。
- 使用 HTTP 的 GET、POST、PUT、DELETE 等方法操作资源。
- URL 中尽量使用名词而不是动词。
- 无状态：服务端不保存客户端状态，客户端每次请求携带状态信息。

PPT 例子：

- 可以：`GET /zoos/ID`
- 不推荐：`POST /accounts/1/transfer/500/to/2`

## 5. 面向 UI 的功能测试

UI 功能测试重点：

- 按功能、子功能、功能点分层展开。
- 基于输入域、组合测试等方法设计输入。
- 扮演用户角色，遍历主要操作路径。
- 针对不同设置进行测试。
- 检查异常操作和错误提示。

UI 测试不能只看界面漂不漂亮，更要验证功能逻辑和用户场景。

## 6. 功能测试自动化

PPT 中功能测试自动化分为：

| 层次 | 特点 | 工具例子 |
| --- | --- | --- |
| 单元测试自动化 | 更容易实现，投资回报率高 | JUnit |
| 接口自动化 | 适合集成测试或系统测试阶段，需要维护用例、环境和数据 | Postman |
| Web UI 自动化 | 贴近用户界面，但维护成本高，对弹窗等异常情况处理困难 | Selenium、Cypress |
| Android UI 自动化 | 移动端界面操作自动化 | Appium、Espresso、Robotium |

自动化测试不能完全替代人工测试，有些结果仍需要人工判断。

## 7. Postman

Postman 是 API 平台，可用于使用、构建和测试 API。

PPT 重点：

- 可创建并发送 HTTP 请求。
- 可发送 XML、SOAP 请求。
- 可传递参数。
- 可创建 collection 保存测试请求。
- pre-request script 在请求发送前执行。
- post-response script 在响应收到后执行。
- 可使用 `pm.response`、`pm.response.test`、`pm.expect` 编写断言。

## 8. Web UI 自动化

### 8.1 Selenium

Selenium 是 Web 应用自动化测试框架。

主要组件：

| 组件 | 作用 |
| --- | --- |
| Selenium WebDriver | 核心组件，向浏览器发送指令并执行 |
| Selenium IDE | 浏览器插件形式的集成开发环境 |
| Selenium Grid | 远端机器上并行运行浏览器实例 |

基本流程：

1. 创建 WebDriver 实例。
2. 打开网页。
3. 执行动作。
4. 断言结果。

### 8.2 Cypress

Cypress 是基于 JavaScript 的前端测试工具。

特点：

- 可测试浏览器中运行的内容。
- 安装后可快速创建、编写、运行测试。
- 支持每一步操作回看。
- 可写端到端测试、集成测试、单元测试。
- 测试代码和应用程序运行在 Cypress 控制的浏览器中。

## 9. Android App UI 自动化

PPT 工具例子：

- Robotium。
- UIAutomator。
- MonkeyRunner。
- Appium。
- Espresso。
- Selendroid。
- TestComplete。

两类机制：

| 机制 | 特点 |
| --- | --- |
| UIAutomator/Appium | 测试进程与目标应用进程分开，只能模拟触发事件 |
| Instrumentation/Robotium/Espresso/Selendroid | 测试进程和目标应用在同一进程，可更精确控制应用 |

Instrumentation 框架能力：

- 在主程序启动前创建模拟系统对象。
- 控制应用生命周期。
- 发送 UI 事件。
- 执行期间检查程序状态。

## 10. 回归测试

回归测试在已经开发并测试过的软件发生改变后进行，对软件再次进行功能或非功能测试，以保证程序行为仍与预期相符。

可能触发回归测试的改变：

- 修改 bug。
- 软件增强。
- 配置改变。
- 重要部件更换。

回归缺陷：

> 原来正常工作的功能，在需求没有变化的情况下，由于其他改动影响而产生的问题。

回归测试目标：

> 发现回归缺陷。

自动化测试对回归测试很重要，因为完整手工回归成本很高。

### 10.1 回归测试策略

每次回归都重跑所有用例通常不现实，需要选择测试用例。

| 策略 | 含义 |
| --- | --- |
| 再测试所有测试用例 | 最全面，但成本最高 |
| 基于风险选择测试用例 | 优先测试风险高、影响大的功能 |
| 基于操作剖面选择测试用例 | 优先测试用户最常使用的场景 |
| 再测试修改部分 | 重点测试被修改部分及其影响范围 |

## 11. 精准测试

精准测试建立软件功能、代码、测试用例之间的映射关系。

回归测试中的精准测试思路：

1. 比较新旧代码差异。
2. 定位被改动的代码。
3. 分析受影响的代码。
4. 根据代码和测试用例映射选择回归测试用例。

```mermaid
flowchart LR
  A["新旧代码差异"] --> B["定位改动代码"]
  B --> C["分析影响范围"]
  C --> D["代码-用例映射"]
  D --> E["选择回归测试用例"]
```

价值：

- 减少不必要的回归测试。
- 更快定位受影响功能。
- 提高回归测试效率。

风险：

- 映射关系不准确会漏测。
- 需要持续维护代码、功能、用例之间的关系。

## 12. 本章速记

1. 系统测试在集成测试后进行，验证完整系统与需求一致。
2. 系统功能测试关注功能、UI、数据、逻辑、接口。
3. 接口测试不只测接口通不通，更要通过接口验证系统功能。
4. RESTful 强调资源、URI、HTTP 方法、无状态。
5. 自动化测试分层：单元更容易，接口收益较高，UI 维护成本较高。
6. 回归测试用于发现修改引入的回归缺陷。
7. 精准测试通过代码和测试用例映射选择回归用例。
