# 第5章：单元测试与集成测试

本章按 PPT 整理为三条线：==代码评审与静态分析==、==单元测试==、==集成测试==。考试重点是概念、对比、任务、驱动/桩、集成策略。

## 1. 本章考点地图

| 模块 | 必须掌握 |
| --- | --- |
| 代码评审 | code review、peer review、walk through、inspection |
| 代码评审价值 | 可发现大量代码缺陷，使用 checklist 提升效果 |
| 走查与会议审查 | 准备、形式、人员、方法、文档区别 |
| 静态检测工具 | FindBugs、PMD、Checkstyle、SonarLint 等了解 |
| 单元测试目标 | 函数、类、模块等最小单元；规范性、正确性、安全性、性能 |
| 单元测试任务 | 路径、局部数据结构、接口、边界、容错、内存 |
| 驱动程序和桩程序 | driver 模拟上级模块，stub 模拟下层模块 |
| JUnit、Mock、JaCoCo | 了解工具作用 |
| 集成测试 | 大棒、自顶向下、自底向上、三明治、微服务契约测试 |
| CI/CD | 持续集成、持续交付、持续部署、持续测试 |

## 2. 代码评审与分析

### 2.1 编码标准和规范

编码标准和规范能增加程序的可读性、可靠性、可维护性和可移植性。

PPT 例子：

- MISRA C coding standard。
- Google C++ Style Guide。
- Google Java Style Guide。

考试理解：编码规范不是形式主义，它可以减少隐蔽缺陷和维护成本。

### 2.2 代码评审价值

PPT 提到：代码审查是一种有效测试方法，统计上可发现代码中 60% 以上的缺陷。

常见形式：

| 形式 | 含义 |
| --- | --- |
| peer review | 代码互查 |
| walk through | 代码走查 |
| inspection | 会议审查 |

实践要点：

- 一次检查约 200-400 行代码。
- 合适速度约 300-500 LOC/hour。
- 审查前作者应注释代码。
- 使用 checklist 可以提升评审效果。
- 限时，避免跑题。
- 发现问题要记录，避免现场直接修改。

## 3. 走查与会议审查

| 对比项 | 走查 | 会议审查 |
| --- | --- | --- |
| 准备 | 成员通读设计和材料 | 准备需求、设计、源代码、编码标准、缺陷检查表 |
| 形式 | 相对非正式会议 | 正式会议 |
| 参加人员 | 开发人员为主 | 项目组成员，可包括测试人员 |
| 技术方法 | 讲解、讨论、模拟运行 | 按缺陷检查表逐项检查 |
| 注意事项 | 限时，不现场修改代码 | 限时，不现场修改代码 |
| 生成文档 | 会议记录 | 静态分析错误报告、评审报告 |
| 目标 | 检查代码是否符合规范、有无逻辑错误 | 检查代码是否符合规范、有无逻辑错误 |

## 4. 代码评审典型问题

PPT 案例中出现的典型问题：

| 问题 | 说明 |
| --- | --- |
| 空指针保护错误 | 字符串比较时可把常量放前面，如 `"schadmin".equals(userName)` |
| 数据类型转换错误 | 需要判断转换参数类型和异常 |
| 字符串或数组越界 | 访问数组前要判断长度 |
| 资源未关闭 | 流使用结束后必须关闭 |
| synchronized 使用不当 | 可能造成并发性能问题 |

## 5. 静态检测工具

| 语言/领域 | 工具例子 |
| --- | --- |
| Java | FindBugs、PMD、Checkstyle、SonarLint |
| C/C++ | Coverity Static Analysis、CxSAST、Flawfinder |
| 安全检测 | SAST、OWASP 相关工具 |

代码分析工具常见使用场景：

- IDE 中实时分析。
- 每次代码提交时分析。
- 与 CI/CD 流水线集成。
- 检查 SQL 注入、跨站脚本、资源泄漏、硬编码凭据、配置问题等。

## 6. 单元测试

### 6.1 为什么要进行单元测试

PPT 给出的理由：

- 编程和编译运行后，每 1000 行代码可能残留若干 bug。
- 寻找和修改错误的代价占总体开发投资很大比例。
- 缺陷发现越早，修复成本越低。
- 单元测试中发现问题更容易，修正也更容易。
- 单元质量是系统质量的基石。
- 单元测试比系统测试更容易测得彻底。

### 6.2 单元测试的其他价值

| 价值 | 说明 |
| --- | --- |
| 支持扩展 | 大系统需要基础测试支撑扩展 |
| 引导更好设计 | 可测试代码通常参数更少、依赖更清晰 |
| 支持变化 | 修改代码后可快速验证是否破坏旧行为 |
| 避免回归缺陷 | 单元测试形成安全网 |
| 节省测试时间 | 单元测试便宜、快速、易自动化 |
| 记录代码行为 | 单元测试也是被测代码行为示例 |

## 7. 单元测试目标和要求

单元测试对象是软件设计的最小单位：函数、类、模块、组件。

单元测试关注：

- 代码规范性。
- 功能正确性。
- 安全性。
- 性能。
- 数据是否正确流入流出单元。
- 单元内部数据能否保持完整。
- 边界数据能否正常工作。
- 是否达到覆盖率要求。

单元测试依据：

- 软件需求规格说明书。
- 软件详细设计说明书。

通过的一般标准：

- 功能与设计需求一致。
- 接口与设计需求一致。
- 达到相关覆盖率要求。

## 8. 单元测试六项任务

| 任务 | 检查重点 |
| --- | --- |
| 模块独立执行路径测试 | 保证重要路径被执行，至少覆盖语句；检查算符优先级、变量初始化、精度、表达式、逻辑运算 |
| 局部数据结构测试 | 检查数据在执行过程中的正确性和完整性；类型、初值、变量名、上溢下溢等 |
| 模块接口测试 | 检查输入输出参数、调用参数、文件属性、只读参数等 |
| 单元边界条件测试 | 合法/非法数据、边界内/边界外数据 |
| 单元容错测试 | 异常处理、错误提示、错误记录、定位信息 |
| 内存分析 | 内存泄漏、内存分配错误、上下文定位 |

## 9. 驱动程序与桩程序

| 概念 | 含义 | 作用 |
| --- | --- | --- |
| driver 驱动程序 | 模拟被测模块的上级模块 | 调用被测模块，使其能独立运行 |
| stub 桩程序 | 模拟被测模块调用的下层模块 | 替代尚未完成或不想真实调用的依赖 |

```mermaid
flowchart LR
  A["driver<br/>上级模拟"] --> B["Function under test<br/>被测函数/模块"]
  B --> C["stub<br/>下层模拟"]
```

为什么需要：

- 隔离被测单元。
- 控制输入和依赖行为。
- 让未集成的单元也能执行测试。

## 10. 类的单元测试

类的单元测试可以看作对类成员函数进行测试。

注意点：

- 核心或重要方法要全面测试。
- 黑盒和白盒方法都可用于类方法。
- 要考虑继承和多态对测试的影响。
- 复杂情况下可能采用展平测试。

## 11. JUnit、Mock 与 JaCoCo

### 11.1 JUnit

JUnit 是 Java 单元测试框架。PPT 中强调：

- Assert 用于判断实际结果与预期结果是否一致。
- Annotation 用于标注测试方法和生命周期方法，如 `@Test`、`@BeforeEach`、`@AfterEach` 等。

### 11.2 Stub、Mock、Fake

| 类型 | 特点 |
| --- | --- |
| Stub | 占位并返回预定义数据，常用于替代真实依赖 |
| Mock | 可记录调用，并在断言中验证预期动作是否发生 |
| Fake | 有可工作的简化实现，但不同于生产实现 |

关键区别：

- Stub 主要提供数据。
- Mock 还能验证交互。
- Fake 是简化版真实实现。

### 11.3 Mockito

Mockito 支持：

- `mock()` 或 `@Mock` 创建 mock。
- `when()` 或 `given()` 指定行为。
- `spy()` 或 `@Spy` 部分模拟真实对象。
- `@InjectMocks` 自动注入依赖。
- `verify()` 检查方法是否按参数、次数、顺序调用。

### 11.4 JaCoCo

JaCoCo 是 Java 覆盖率统计工具。

覆盖率分析维度：

- 指令。
- 分支。
- 行。
- 方法。
- 类。
- 圈复杂度。

## 12. 集成测试

集成测试把已经分别通过测试的单元按设计要求集成起来，检查单元之间的接口问题。

常见接口问题：

- 参数不匹配。
- 参数传递错误。
- 数据丢失。
- 调用顺序错误。
- 模块间约定不一致。

## 13. 单体架构集成模式

### 13.1 非渐增式：大棒集成

先分别测试每个模块，然后一次性把所有模块集成。

优点：简单。

缺点：一旦出错，很难定位真正出错的位置、模块和原因。不推荐大系统使用，只适合规模较小系统。

### 13.2 渐增式集成

每次把下一个要测试的模块与已测模块结合，测试后再加入下一个模块。

主要形式：

- 自顶向下。
- 自底向上。
- 混合策略。

### 13.3 自顶向下集成

从主控模块开始，沿控制结构向下移动。

步骤：

1. 测试主控模块，用桩代替下层模块。
2. 每次用一个实际模块替代一个桩。
3. 加入新单元时进行测试，必要时做回归测试。
4. 重复直到所有单元加入。

优点：较早发现上层重要模块问题。

缺点：需要大量桩；底层模块测试较晚。

### 13.4 自底向上集成

从底层模块开始。

步骤：

1. 从底层单元开始，可把多个底层单元合并测试。
2. 编写驱动程序。
3. 去掉驱动，向上组合成更大功能族。
4. 直至所有单元完成。

优点：底层基础模块测试充分。

缺点：上层控制逻辑较晚测试，需要驱动程序。

### 13.5 三明治式集成

三明治式集成结合自顶向下和自底向上：

- 界限以上用自顶向下。
- 界限以下用自底向上。
- 两边向中间集成。

优点：结合两种方法，减少部分桩程序。

缺点：真正集成前，每个独立模块可能没有完全测试。

改进的三明治方法：两头向中间集成，同时保证每个模块得到单独测试。

## 14. 微服务架构集成测试

微服务集成测试关注：

| 内容 | 目的 |
| --- | --- |
| 外部通信测试 | 确认通信是否通畅，发现协议层错误，如 HTTP 头丢失、SSL 使用错误、请求/响应不匹配 |
| 数据库访问测试 | 检查数据结构与数据库是否相符，ORM 映射和网络错误处理是否正确 |
| 核心功能测试 | 验证关键业务链路 |

## 15. 契约测试

消费者驱动的契约测试（CDC testing）：

- 契约是服务消费者与提供者之间的交互约定。
- 包括请求和响应的数据格式。
- 从消费者需求出发设计测试用例并产生契约。
- 再验证 provider 端是否满足契约。

优势：

- 降低服务集成难度。
- 把联调拆成单元测试和接口测试。
- 团队可离线验证。
- 接口变更有迹可循。

PPT 工具例子：Pact、Pacto、Spring Cloud Contract 等。

## 16. CI/CD 与持续测试

| 概念 | 含义 |
| --- | --- |
| CI 持续集成 | 提交代码后自动测试并合并分支 |
| CD 持续交付 | 自动测试并产生可发布版本 |
| 持续部署 | 持续交付后加入自动部署环节 |
| 持续测试 | 自动化和持续监控贯穿开发周期 |

PPT 关系：持续部署倒逼持续测试，持续测试支撑持续交付和持续部署。

## 17. 本章速记

1. 代码评审可发现大量缺陷，checklist 很重要。
2. 走查较非正式，会议审查更正式。
3. 单元测试对象是函数、类、模块等最小单元。
4. 单元测试六任务：路径、局部数据、接口、边界、容错、内存。
5. driver 模拟上级，stub 模拟下层。
6. Stub 返回数据，Mock 验证交互，Fake 是简化实现。
7. 集成测试重点找接口问题。
8. 大棒集成不利于定位问题；自顶向下用桩，自底向上用驱动。
9. 微服务集成测试可用消费者驱动契约测试。
