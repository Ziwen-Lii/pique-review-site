# 第2章：软件测试基本概念

本章是术语章，选择题和简答题很多。按 PPT 重点复习：==软件质量与软件缺陷==、==测试分类==、==静态/动态测试==、==黑盒/白盒测试==、==V&V==、==测试层次==、==测试工作范畴==。

## 2. 软件质量与软件缺陷

PPT 的核心句：

> 缺陷是质量的对立面。违背质量、违背客户意愿、不能满足客户要求，就会引起缺陷或产生缺陷。

### 2.1 软件质量定义

软件质量可以理解为：软件产品满足明示需求和隐含需求的能力。

| 质量类型 | 含义 | 例子 |
| --- | --- | --- |
| 内部质量 | 开发者能看到、影响开发过程的属性 | 代码规范性、耦合度、复杂度、可维护性 |
| 外部质量 | 用户和测试者能感受到的软件属性 | 正确、可靠、易用、响应快 |
| 使用质量 | 产品在真实使用过程中表现出来的质量 | 操作方便、界面友好、任务完成率高 |

PPT 提醒：

> 项目工期紧时，外部质量可以被迫妥协，内部质量不容轻易妥协。内部质量坏了，会长期提高维护和测试成本。

### 2.2 ISO 25010 八个产品质量特性

| 特性 | 中文解释 | 常见测试关注点 |
| --- | --- | --- |
| 功能适应性 | 功能是否满足需求 | 功能测试、验收测试 |
| 性能效率 | 响应时间、吞吐量、资源利用是否合理 | 性能测试 |
| 兼容性 | 软件与硬件、系统、浏览器、数据格式是否兼容 | 兼容性测试 |
| 易用性 | 用户是否容易理解、学习和操作 | 易用性测试、A/B 测试 |
| 可靠性 | 规定条件下持续运行的能力 | 可靠性测试、容错测试 |
| 安全性 | 保密性、完整性、可用性等是否满足要求 | 安全测试 |
| 可维护性 | 是否容易分析、修改、测试 | 代码评审、静态分析 |
| 可移植性 | 是否容易迁移到其他环境 | 安装测试、移植测试 |

使用质量模型还包括有效性、效率、满意度、风险控制、语境覆盖等内容。

## 3. 软件缺陷

### 3.1 缺陷定义

从内部看，软件缺陷是软件产品开发或维护过程中存在的错误、毛病等问题。

从外部看，软件缺陷是系统所需要实现的某种功能失效或被违背。

### 3.2 满足以下之一即可认为是缺陷

| 准则 | 例子 |
| --- | --- |
| 未实现需求说明书要求的功能 | 需求要求能导出报表，但系统没有导出功能 |
| 出现需求说明书指明不应出现的错误 | 合法输入导致程序崩溃 |
| 实现了需求说明书未提到的功能 | 多出一个未授权入口 |
| 未实现虽未明确提及但应该实现的功能 | 密码框明文显示密码 |
| 用户体验不佳 | 难以理解、不易使用、运行缓慢 |

### 3.3 缺陷类型

| 类型 | 说明 |
| --- | --- |
| 输入缺陷 | 输入范围、格式、异常输入处理错误 |
| 输出缺陷 | 输出值、输出格式、提示信息错误 |
| 计算缺陷 | 公式、精度、类型转换、溢出错误 |
| 接口缺陷 | 参数个数、类型、顺序、协议不匹配 |
| 数据缺陷 | 数据结构、持久化、一致性、迁移错误 |
| 逻辑缺陷 | 分支条件、业务流程、状态转换错误 |

### 3.4 缺陷产生原因

- 需求规格说明书问题：理解错误、不完整、不一致、变化频繁。
- 团队协作问题：理解不一致、职责划分不清、不重视文档。
- 应用场景考虑不足：高并发、海量数据、软件依赖、异常环境。
- 技术问题：技术掌握不足、系统过于复杂、算法和并发错误。

PPT 提醒：需求和设计阶段往往是缺陷出现较多的地方。越早通过评审发现缺陷，修复成本越低。

## 4. test oracle 测试预言

test oracle 是判断被测系统实际输出是否正确的依据。

常见依据：

| 来源 | 说明 |
| --- | --- |
| 需求规格说明书 | 最直接、最标准的判断依据 |
| 设计规范文档 | 判断接口、结构、约束、内部行为 |
| 启发式测试预言 | 根据经验判断某结果明显不合理 |
| 基于模型的测试预言 | 通过模型推导期望输出或状态 |
| 一致性预言 | 不同版本、不同实现、同类系统之间应保持一致 |

考试句：

> 没有 test oracle，就难以判断测试是否通过。

## 5. 软件测试分类

### 5.1 按测试对象分类

| 层次 | 对象 | 目标 |
| --- | --- | --- |
| 单元测试 | 函数、类、模块、组件 | 验证最小设计单元是否正确 |
| 集成测试 | 单元之间接口 | 发现接口、参数、调用顺序、数据传递问题 |
| 系统测试 | 完整系统和运行环境 | 验证系统与需求规格说明书一致 |
| 验收测试 | 用户业务和接受标准 | 验证系统是否满足用户合理期待 |

### 5.2 按测试目的分类

| 类型 | 目标 |
| --- | --- |
| 功能测试 | 功能是否按需求工作 |
| 性能测试 | 响应时间、吞吐量、资源利用是否达标 |
| 可靠性测试 | 长时间运行、容错、恢复能力 |
| 安全性测试 | 认证、授权、漏洞、攻击防护 |
| 兼容性测试 | 平台、浏览器、硬件、数据格式兼容 |
| 回归测试 | 修改后原有功能是否被破坏 |

### 5.3 按是否执行程序分类

| 类型 | 是否执行程序 | 典型活动 |
| --- | --- | --- |
| 静态测试 | 不执行程序 | 需求评审、设计评审、代码评审、静态分析 |
| 动态测试 | 执行程序 | 输入数据，观察输出、状态和行为 |

### 5.4 按是否关注内部结构分类

| 对比项 | 黑盒测试 | 白盒测试 |
| --- | --- | --- |
| 是否了解代码 | 不关注内部结构 | 已知代码和控制结构 |
| 依据 | 需求、规格说明、用户场景 | 源代码、控制流、数据流 |
| 主要思路 | 由输入输出验证功能 | 覆盖语句、分支、条件、路径 |
| 优点 | 接近用户视角 | 能发现内部路径和逻辑问题 |
| 局限 | 可能漏掉内部结构错误 | 需要代码知识，成本较高 |

### 5.5 其他分类

| 对比 | 含义 |
| --- | --- |
| 主动测试 vs 被动测试 | 主动测试由测试人员操作系统；被动测试监控真实运行数据 |
| 手工测试 vs 自动化测试 | 手工执行 vs 工具或脚本执行 |
| 基于脚本测试 vs 探索式测试 | 先设计脚本后执行 vs 边学习、边设计、边测试、边分析 |

## 6. 静态测试

静态测试包括两大类：产品评审和静态分析。

注意：

> 管理评审、流程评审属于质量保证；需求、设计、代码等产品工件的评审才属于静态测试。

### 6.1 评审形式

| 形式 | 特点 |
| --- | --- |
| 同伴评审 | 较轻量，成员互查 |
| 走查 | 作者带领成员通读材料，讲解、讨论、模拟运行 |
| 会议审查 | 较正式，有角色、检查表、记录和评审报告 |

### 6.2 需求评审看什么

- 需求是否正确、完整、一致。
- 描述是否清楚，是否可测试。
- 是否遗漏需求点。
- 是否存在多余或没意义的需求。
- 不同参与方理解是否一致。

### 6.3 设计评审看什么

- 系统架构是否合理。
- 是否存在单点失效。
- 数据是否完整、一致。
- 是否具有可测试性。
- 功能设计是否完整覆盖需求。

### 6.4 静态分析

静态分析是在不运行代码的情况下研读或分析源代码。

| 类型 | 说明 |
| --- | --- |
| 人工检测 | 偏重编码风格、质量、逻辑设计和编码错误 |
| 工具辅助 | 通过词法分析、语法分析、控制流分析、数据流分析等检查可疑结构 |

PPT 中出现的相关术语：AST、IR、Token、符号执行、抽象解释。

## 7. Verification 与 Validation

| 概念 | PPT 问法 | 中文含义 |
| --- | --- | --- |
| Verification | Are we building the product right? | 是否正确地构造了软件，是否符合规格说明 |
| Validation | Are we building the right product? | 是否构造了用户真正需要的软件 |

简单区分：

- Verification 关注“做得对不对”。
- Validation 关注“做的是不是用户真正要的”。

## 8. 软件测试层次

| 层次 | 主要任务 | 常见问题 |
| --- | --- | --- |
| 单元测试 | 测试函数、类、模块、组件 | 内部逻辑错误、边界错误、局部数据错误 |
| 集成测试 | 测试单元之间接口 | 参数不匹配、数据丢失、调用顺序错误 |
| 系统测试 | 测试完整系统 | 功能、安全、性能、可靠性、易用性等 |
| 验收测试 | 面向用户业务验收 | 是否满足用户需求和可接受性 |

单元测试一般由程序员为主完成；验收测试面向未来用户。α 测试和 β 测试属于验收测试相关形式。

## 9. 软件测试工作范畴

PPT 把测试工作分为：

```mermaid
flowchart LR
  A["测试分析"] --> B["测试策略制订"]
  B --> C["测试计划"]
  C --> D["测试设计"]
  D --> E["测试执行"]
  E --> F["测试结果和过程评估"]
```

| 阶段 | 主要内容 |
| --- | --- |
| 测试分析 | 明确测试范围、测试目标、优先级和测试任务 |
| 测试策略 | 选择测试方式、方法、技术、过程安排 |
| 测试计划 | 目标范围、估算、风险计划、进度、资源、跟踪控制 |
| 测试设计 | 总体设计、详细用例设计、测试结构设计 |
| 测试执行 | 手工执行、自动化执行、探索式测试 |
| 结果和过程评估 | 分析覆盖率、缺陷趋势、缺陷分布、计划执行效果 |

## 10. 测试用例

PPT 定义：

> 测试用例是为了特定测试目的而设计的测试条件、测试数据以及相关操作等构成的一个特定使用场景。

测试用例的价值：

- 是典型场景建模。
- 可以复用。
- 是测试知识积累的过程。

## 11. 基于脚本的测试与探索式测试

| 对比项 | 基于脚本的测试 | 探索式测试 |
| --- | --- | --- |
| 节奏 | 先设计、后执行 | 学习、设计、执行、分析并行 |
| 关注 | 需求、测试文档、明确标准 | 产品交互、上下文、人的能力 |
| 优点 | 严谨、规范、可控、适合回归 | 灵活、适应变化、容易发现未知问题 |
| 风险 | 可能僵化，难发现需求外问题 | 依赖测试人员经验，记录和复现要求高 |

## 12. 本章速记

1. 质量是满足明示和隐含需求的能力，缺陷是质量的对立面。
2. test oracle 是判断测试是否通过的依据。
3. 静态测试不执行程序，动态测试执行程序。
4. 管理评审不是静态测试，而是质量保证。
5. Verification：是否正确构造软件；Validation：是否构造用户真正需要的软件。
6. 四个测试层次：单元、集成、系统、验收。
7. 测试工作范畴：分析、策略、计划、设计、执行、评估。
