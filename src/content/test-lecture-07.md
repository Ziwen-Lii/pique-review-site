## 2. 性能测试

### 2.1 为什么要做性能测试

PPT 强调：在 SaaS、万物互联、高并发环境下，性能问题非常常见。

性能问题表现：

- CPU 使用率达到 100%。
- 内存泄漏导致资源耗尽。
- 线程、GDI、数据库连接等资源瓶颈。
- 启动系统、打开页面越来越慢。
- 查询列表等待很久。
- 网络下载速度很低。

### 2.2 性能测试定义

性能测试是为了发现系统性能问题或获取系统性能指标而进行的测试。

通常做法：

1. 在真实或接近真实环境中。
2. 在特定负载条件下。
3. 用工具模拟实际软件系统运行和用户操作。
4. 监控性能指标。
5. 分析结果，判断性能状况。

### 2.3 性能测试目标

| 目标 | 说明 |
| --- | --- |
| 获取性能指标 | 得到响应时间、吞吐量、资源利用率等数据 |
| 验证性能需求 | 判断系统是否达到用户提出的性能指标 |
| 发现性能瓶颈 | 找到制约性能的资源、代码、架构或环境问题 |
| 支持性能优化 | 为调优和容量规划提供依据 |

## 3. 负载与负载模式

PPT 中负载相关概念：

| 概念 | 含义 |
| --- | --- |
| RPS | 每秒请求数 |
| SC | 并发连接数 |
| thinking time | 用户两次请求之间的间隔时间 |
| concurrent users | 并发用户数，与 RPS、SC、thinking time 有关 |

思考时间很重要。如果把请求间隔设为 0，服务器会立即承受不真实的高压，形成队列甚至内存耗尽。

负载模式包括：

- 循环次数或持续时间。
- 均匀加载。
- 峰值交替加载。
- 高低突变加载。

## 4. 性能测试类型

| 类型 | 含义 |
| --- | --- |
| 性能基准测试 | 在标准配置下获得性能指标，作为以后改进基线 |
| 性能验证测试 | 验证系统是否达到已定义的性能指标 |
| 性能规划测试 | 在不同环境配置下获取指标，支持软硬件选型 |
| 容量测试 | 获取系统可承载容量，可看作性能测试的一种 |
| 压力测试 | 模拟高负载、异常负载、长时间运行，加速系统崩溃，找瓶颈和不稳定点 |
| 渗入测试 soak test | 长时间运行，让内存泄漏、GC、资源问题逐渐暴露 |
| 峰谷测试 peak-rest test | 高负载和低负载交替，发现隐藏性能问题 |

注意：这里的“渗入测试”是性能测试中的 soak test，不是安全测试里的渗透测试。

## 5. 全生命周期性能测试

PPT 强调性能测试不是上线前才做，而应贯穿生命周期：

| 阶段 | 性能活动 |
| --- | --- |
| 需求 | 明确性能需求、业务场景、业务数据 |
| 架构 | 架构评估、性能建模、容量规划 |
| 设计/实现 | 代码评审、单元性能测试、代码优化 |
| 测试 | 设计并执行性能测试，定位瓶颈 |
| 在线 | 性能监控、健康检查、容量管理、持续优化 |

性能需求要可测试。比如“每天处理一千万笔交易”不够直接，要换算成平均 TPS、高峰 TPS 等量化指标。

## 6. 性能测试工具 JMeter

PPT 中重点工具：JMeter。

### 6.1 JMeter 组成

| 组件 | 作用 |
| --- | --- |
| 测试计划 | 管理整个测试脚本 |
| 线程组 | 模拟用户和并发 |
| 采样器 | 发送 HTTP、JDBC、FTP 等请求 |
| 逻辑控制器 | 控制请求执行顺序 |
| 断言 | 判断响应是否符合预期 |
| 前置/后置处理器 | 请求前后处理 |
| 监听器 | 查看结果、响应时间、吞吐量等 |
| 定时器 | 控制请求间隔 |

### 6.2 JMeter 常用能力

- 支持 HTTP、HTTPS、SOAP、REST、FTP、JDBC 等。
- 支持非 GUI 命令行运行。
- 支持录制脚本。
- 支持分布式压测。
- 支持插件扩展。

## 7. 安全性测试

### 7.1 漏洞与安全缺陷

漏洞也称脆弱性，是系统中可能被威胁利用并造成损害的薄弱环节。

漏洞来源：

| 来源 | 例子 |
| --- | --- |
| 恶意 | 木马病毒、陷门、逻辑炸弹、定时炸弹 |
| 非恶意 | 校验错误、范围错误、认证错误、边界条件错误、逻辑错误 |

### 7.2 攻击带来的安全问题

- 信息泄露。
- 破坏完整性。
- 拒绝服务 DoS/DDoS。
- 非授权访问。
- 窃听。
- 业务数据流分析。
- 假冒。
- 旁路控制。
- 授权侵犯。
- 抵赖。
- 病毒和恶意软件。

### 7.3 STRIDE 威胁模型

PPT 中写作 STRIED，但内容对应常见 STRIDE：

| 字母 | 含义 | 中文 |
| --- | --- | --- |
| S | Spoofing | 假冒身份 |
| T | Tampering | 篡改 |
| R | Repudiation | 抵赖 |
| I | Information Disclosure | 信息泄露 |
| D | Denial of Service | 拒绝服务 |
| E | Elevation of Privilege | 特权提升 |

### 7.4 常见漏洞模式

- 缓冲区溢出。
- 图像格式漏洞。
- 竞争条件漏洞。
- SQL 注入。
- XSS。
- CSRF。
- Frame-proxy 攻击。
- 并发漏洞。

OWASP Top 10 是 Web 安全常见漏洞清单。

## 8. 安全目标与安全需求

### 8.1 CIA 三元组

| 目标 | 含义 |
| --- | --- |
| 保密性 | 信息只被授权者访问 |
| 完整性 | 信息和处理方法保持准确、原始、一致 |
| 可用性 | 授权用户需要时可以访问信息 |

安全目标常互相冲突，需要平衡。例如完全禁止访问可提高保密性，但会破坏可用性。

### 8.2 其他安全要求

- 真实性。
- 不可抵赖性。
- 可追溯性。
- 可控制性。
- 可审查性。

### 8.3 软件安全性需求

| 类型 | 内容 |
| --- | --- |
| 数据安全性 | 加密、安全存储、安全访问、认证、备份 |
| 系统安全性 | 身份认证、权限限制、异常处理、入口验证、系统强壮性设计 |

## 9. 安全性测试范围与方法

### 9.1 两类安全测试

| 类型 | 目标 |
| --- | --- |
| 安全功能测试 | 测试数据机密性、完整性、可用性、身份认证、授权、访问控制、审计、安全管理等功能 |
| 安全漏洞测试 | 从攻击者角度发现设计、实现、操作、管理中的可利用缺陷 |

功能性测试与安全性测试区别：

- 功能性测试：软件做它应该做的事。
- 安全性测试：软件不做它不应该做的事。

### 9.2 安全性测试任务

- 检查需求规格中防止危险状态措施是否有效。
- 对设计中安全结构、算法、容错、冗余、中断处理进行针对性测试。
- 在异常条件下测试软件是否进入不安全状态。
- 对安全关键单元和组件加强测试。

### 9.3 安全测试方法

| 方法 | 说明 |
| --- | --- |
| 静态测试 | 工具扫描、代码分析、架构/代码人工评审、安全风险评估 |
| 动态功能测试 | 身份认证、访问控制、规则验证、负面测试、传输检查、存储检查 |
| 模糊测试 | 输入随机或变异数据，观察崩溃、异常退出等问题 |
| 渗透测试 | 模拟攻击者发现安全漏洞 |
| 基于威胁的方法 | 从外部识别威胁并测试是否可发生 |
| 基于漏洞的方法 | 从内部识别漏洞，常借助扫描工具 |

### 9.4 模糊测试

模糊测试是在被测程序中附加随机数据或变异数据作为输入，如果程序崩溃或异常退出，就可能定位缺陷。

模糊器类型：

| 类型 | 含义 |
| --- | --- |
| 黑盒随机模糊 | 对正确格式输入进行随机变异 |
| 基于语法的模糊 | 指定输入格式语法，对复杂格式输入进行变异 |
| 白盒模糊 | 动态执行程序，收集分支约束，再用求解器生成新输入 |

## 10. 兼容性测试

兼容性测试覆盖三类：

| 类型 | 内容 |
| --- | --- |
| 硬件兼容 | CPU、内存、设备、移动端型号等 |
| 软件兼容 | 操作系统、浏览器、中间件、数据库、第三方软件 |
| 数据兼容 | 不同版本、不同格式、历史数据、迁移数据 |

PPT 小结强调：数据兼容更重要，因为数据更具价值。系统升级后，历史数据能否正确读取、迁移和解释，往往比界面兼容更关键。

## 11. 可靠性测试与混沌工程

可靠性测试关注系统在规定条件下、规定时间内完成规定功能的能力。

常见关注：

- 长时间运行是否稳定。
- 故障后能否恢复。
- 依赖异常时是否降级。
- 资源压力下是否保持服务。

### 11.1 故障注入

故障注入通过人为制造故障，观察系统能否发现、隔离、恢复。

常见故障：

- CPU 压力。
- 内存压力。
- 磁盘 I/O 压力。
- 网络延迟。
- 网络断开。
- DNS 故障。
- 杀进程。
- 时间变化。
- 依赖服务失败。

### 11.2 混沌工程

PPT 观点：

> 混沌工程可看作故障注入测试的右移，并提升为工程层次。

混沌工程核心流程：

1. 定义实验场景。
2. 做出稳态假设。
3. 关联故障注入能力。
4. 控制爆炸半径。
5. 运行实验。
6. 观察监控指标。
7. 终止实验并产出报告。
8. 修复问题并验证。

优秀实践：

- 从历史故障中抽象通用故障库。
- 从线下到生产，从小到大逐步推进。
- 强调可观察性。
- 使用开源平台，如 Chaos Mesh、ChaosBlade 等。

## 12. 易用性测试

### 12.1 好 UI 的七个要素

PPT 中良好 UI 元素包括：

1. 符合标准和规范。
2. 直观性。
3. 一致性。
4. 灵活性。
5. 舒适性。
6. 正确性。
7. 实用性。

### 12.2 用户体验测试方法

| 方法 | 含义 |
| --- | --- |
| 探索式测试 | 确定新产品应包含什么功能，评估原型有效性和易用性 |
| 评估性测试 | 发布前或发布后测试最新版本，确保直观使用和良好体验 |
| 比较性测试 | 比较多个产品或设计的易用性，找出更优方案 |

### 12.3 用户体验指标模型

| 模型 | 含义 |
| --- | --- |
| GSM | 从 Goals 出发，转化为 Signal，再建立 Metric |
| PULSE | Page view、Uptime、Latency、Seven-day active user、Earning |
| HEART | Happiness、Engagement、Adoption、Retention、Task success |

## 13. A/B 测试

A/B 测试是将用户分成不同组，同时在线试验产品不同版本，通过真实数据判断哪个方案更好。

### 13.1 A/B 测试特点

| 特点 | 含义 |
| --- | --- |
| 先验性 | 小流量先验证，再推广全流量 |
| 并行性 | 多个版本同时运行，环境一致，节省验证时间 |
| 科学性 | 基于统计数据决策，减少主观经验误判 |

### 13.2 适用场景

- 灰度发布。
- 功能优化。
- 内容优化。
- 运营优化。

### 13.3 核心概念

| 概念 | 含义 |
| --- | --- |
| 场景 | 对应业务场景，场景之间完全独立 |
| 桶 | 圈定特定用户流量，同一场景下不同桶互斥 |
| 层 | 一类实验集合；同层实验互斥，不同层实验正交 |
| 实验 | 验证某功能或策略对指标影响的一部分流量 |
| 流量 | 所有用户请求 |
| 流量正交 | 不同层流量分配相互独立 |

互斥：用户只能进入其中一个实验。

正交：用户进入不同实验之间没有必然关系。

### 13.4 A/B 测试流程

```mermaid
flowchart LR
  A["产生实验想法"] --> B["评估实验优先级"]
  B --> C["设计和开发实验"]
  C --> D["分流运行"]
  D --> E["分析数据"]
  E --> F["应用结果"]
```

### 13.5 A/B 测试平台关注

- 元数据管理：应用、场景、实验创建和配置发布。
- 接入支持：示例代码、分流用户标识测试、实时日志。
- 实时报表和离线报表。
- 异常监控和告警。
- 多级缓存和 SDK 性能优化。
- 分流信息透传，减少业务侵入。

## 14. 本章速记

1. 性能测试目标：获取指标、验证需求、发现瓶颈、支持优化。
2. 性能测试类型：基准、验证、规划、容量、压力、soak、peak-rest。
3. 性能需求必须量化，不能只写“系统要快”。
4. 安全目标核心是 CIA：保密性、完整性、可用性。
5. STRIDE：假冒、篡改、抵赖、信息泄露、拒绝服务、特权提升。
6. 安全功能测试看安全功能是否有效，安全漏洞测试站在攻击者角度找漏洞。
7. 兼容性包含硬件、软件、数据，数据兼容更重要。
8. 混沌工程是故障注入的右移和工程化。
9. A/B 测试用小流量、并行实验和真实数据支持决策。
